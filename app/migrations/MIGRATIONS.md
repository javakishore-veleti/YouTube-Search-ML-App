# Database Migrations

This project uses **[Alembic](https://alembic.sqlalchemy.org/)** for all schema migrations.  
The same codebase works with **SQLite, PostgreSQL, MySQL, and AWS Aurora/RDS** — no code changes needed, just set `DATABASE_URL`.

---

## Folder layout

```
app/                          ← all Python lives here, including migrations
  migrations/
    env.py                    ← reads DATABASE_URL, wired to SQLAlchemy models
    script.py.mako            ← template for new revision files
    MIGRATIONS.md             ← this file
    versions/
      v001_20260322_initial_schema.py    ← first migration (all tables)
      v002_YYYYMMDD_<description>.py     ← future migrations
alembic.ini                   ← config at repo root (script_location → app/migrations)
```

**File naming convention:** `v<rev>_<YYYYMMDD>_<what_changed>.py`  
Example: `v4a2b_20260415_add_tags_to_models.py`  
→ human-readable, date-sorted, tells you *when* and *what* at a glance.

## Database flexibility — just set `DATABASE_URL`

Both `db_engine.py` (app) and `env.py` (Alembic) read from the same env var.  
No code changes. No `alembic.ini` edits. Just set the variable and restart.

| Environment | DATABASE_URL value |
|---|---|
| Local dev (default) | *(not set)* — SQLite at `app/data/models.db` |
| Docker Compose | `postgresql+psycopg2://user:pass@db:5432/appdb` |
| AWS RDS Postgres | `postgresql+psycopg2://user:pass@rds-host:5432/appdb?sslmode=require` |
| AWS Aurora MySQL | `mysql+pymysql://user:pass@aurora-host:3306/appdb` |
| Plain MySQL | `mysql+pymysql://user:pass@host:3306/appdb` |

### Required pip packages per DB driver

```bash
# PostgreSQL / Aurora Postgres / RDS
pip install psycopg2-binary

# MySQL / Aurora MySQL
pip install pymysql
```

Add the relevant package to `requirements.txt` when you switch.

### `.env` example for local Postgres

```dotenv
DATABASE_URL=postgresql+psycopg2://postgres:secret@localhost:5432/vidsage
```

## Common Alembic commands

Run from the **repo root** (where `alembic.ini` lives).

```bash
# Check which revision the DB is on
alembic current

# See full migration history
alembic history --verbose

# Apply all pending migrations  ← also runs automatically on app startup
alembic upgrade head

# Roll back one step
alembic downgrade -1

# Roll back to a specific revision
alembic downgrade v001

# Generate SQL diff without touching the DB (useful for PR review / DBA approval)
alembic upgrade head --sql
```

## How to add a new migration

### Option A — Python (recommended, autogenerated)

1. **Edit the model** in `app/app_common/database/db_models.py`
2. **Auto-generate** the migration:
   ```bash
   alembic revision --autogenerate -m "add_tags_to_models"
   ```
   Creates: `app/migrations/versions/v<hash>_<today>_add_tags_to_models.py`

3. **Review** the generated `upgrade()` and `downgrade()` — always verify before applying.

4. **Apply:**
   ```bash
   alembic upgrade head
   ```

### Option B — Manual Python (for data migrations, backfills, custom logic)

```bash
alembic revision -m "backfill_model_status"
```

Edit the file directly:

```python
from alembic import op
import sqlalchemy as sa

def upgrade() -> None:
    # Use batch_alter_table for SQLite compatibility.
    # On Postgres/MySQL this compiles to a standard ALTER TABLE.
    with op.batch_alter_table("models") as batch_op:
        batch_op.add_column(sa.Column("tags", sa.Text(), nullable=True))

def downgrade() -> None:
    with op.batch_alter_table("models") as batch_op:
        batch_op.drop_column("tags")
```

> **Always use `op.batch_alter_table()`** for any column/index change.  
> SQLite needs it; Postgres/MySQL compile it to a plain `ALTER TABLE` automatically.

## Team rules

| Rule | Why |
|---|---|
| Never edit an already-applied migration | Other devs / prod DB already ran it — create a new one instead |
| Always write `downgrade()` | Enables safe rollbacks without manual SQL |
| One concern per migration file | Easier to review, revert, and cherry-pick |
| Run `alembic current` before creating a new migration | Avoids branched revision chains |
| Commit migration files in the same PR as the model change | Keeps schema and code in sync |
| Use `--sql` in CI to capture the diff for DBA review | Produces plain SQL without touching the DB |
